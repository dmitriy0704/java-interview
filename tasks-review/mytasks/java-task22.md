# Надежность(пулы соединений)



Вы правильно переходите от управления потоками к управлению соединениями с БД — это два взаимосвязанных ресурса. Пулы соединений критически важны для производительности и стабильности любого приложения, работающего с реляционными базами данных.

## 1. Зачем нужен пул соединений?

Создание соединения с БД — очень дорогая операция:
- TCP‑рукопожатие
- Аутентификация в БД
- Выделение ресурсов на сервере БД (сессия, память, курсоры)
- Обычно занимает десятки-сотни миллисекунд

Без пула каждое обращение к БД создавало бы новое соединение → открывало/закрывало → убивало бы производительность приложения и БД.

**Задачи пула:**
- Поддерживать несколько открытых соединений «горячими».
- Переиспользовать их между запросами/транзакциями.
- Ограничивать максимальное число соединений (защита БД от перегрузки).
- Обрабатывать таймауты, валидацию, переподключение при обрыве.

## 2. Как работает пул соединений?

Упрощённая схема:

```
Приложение → вызывает DataSource.getConnection() 
          → пул выдаёт одно из свободных соединений (или создаёт новое, если не достигнут максимум)
          → приложение работает с БД
          → вызывает connection.close()
          → соединение не закрывается физически, а возвращается в пул (становится свободным)
```

Когда приложение «закрывает» соединение, на самом деле оно просто возвращается в пул. Реальное физическое закрытие происходит только при эвикшене (например, соединение долго не использовалось) или при завершении работы пула.

## 3. Основные параметры пула соединений

| Параметр | Что делает | Типичное значение |
|----------|------------|-------------------|
| `maximumPoolSize` | Максимальное число физических соединений, которое может создать пул | 10–30 (зависит от лицензии БД, мощности сервера, числа одновременных запросов) |
| `minimumIdle` | Минимальное количество "простых" соединений, которое пул старается поддерживать | 2–10 |
| `connectionTimeout` | Максимальное время ожидания в очереди при запросе соединения (мс) | 30_000 (30 сек) |
| `idleTimeout` | Время, после которого свободное соединение может быть закрыто (если больше `minimumIdle`) | 600_000 (10 мин) |
| `maxLifetime` | Максимальное время жизни соединения (чтобы избежать проблем с устаревшими сокетами, балансировщиками БД) | 1_800_000 (30 мин) |
| `validationQuery` | Простой запрос для проверки соединения перед выдачей (например, `SELECT 1`) | `SELECT 1` (для MySQL, PostgreSQL) |
| `leakDetectionThreshold` | Время, после которого соединение, не возвращённое в пул, считается утекшим (лог предупреждения) | 60_000 (60 сек) |

## 4. Популярные реализации пулов в Java

| Реализация | Популярность | Особенности | Рекомендация |
|------------|--------------|-------------|---------------|
| **HikariCP** | Очень высокая (де-факто стандарт) | Самый быстрый, лёгкий, минимальное количество багов. Используется в Spring Boot 2+ | **Рекомендуется всегда** |
| **Apache DBCP2** | Средняя | Тяжелее, исторически был популярен, но сейчас медленнее и бажнее HikariCP | Не рекомендуется для новых проектов |
| **Tomcat JDBC Pool** | Высокая (в среде Tomcat) | Хорошая альтернатива, если вы используете Tomcat | Можно использовать, но HikariCP всё равно лучше |
| **Vibur DBCP** | Низкая | Характеристики лучше DBCP, но не дотягивает до Hikari | Не рекомендуется |
| **Oracle UCP** | Только для Oracle | Специфичные фичи для Oracle RAC и AQ | Только если нужны эксклюзивные возможности Oracle |

**Вывод на 2024 год:** используйте **HikariCP**. Он быстр, надёжен, активно поддерживается.

### 5. Пример настройки HikariCP

#### Через конфигурацию в Spring Boot (`application.yml`)

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
    username: appuser
    password: secret
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      leak-detection-threshold: 60000
      pool-name: MyAppHikariPool
      auto-commit: false
      validation-timeout: 5000
      connection-test-query: SELECT 1
```

#### Программная настройка (без Spring)

```java
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:postgresql://localhost:5432/mydb");
config.setUsername("appuser");
config.setPassword("secret");
config.setMaximumPoolSize(20);
config.setMinimumIdle(5);
config.setConnectionTimeout(30000);
config.setIdleTimeout(600000);
config.setMaxLifetime(1800000);
config.setLeakDetectionThreshold(60000);
config.setPoolName("MyHikariPool");
config.setAutoCommit(false);
config.addDataSourceProperty("cachePrepStmts", "true");
config.addDataSourceProperty("prepStmtCacheSize", "250");
config.addDataSourceProperty("prepStmtCacheSqlLimit", "2048");

HikariDataSource dataSource = new HikariDataSource(config);
```

## 6. Связь пула соединений с пулами потоков

Эти два пула **тесно связаны**, и мисматч между ними — частая причина проблем.

**Принцип:** количество потоков, одновременно работающих с БД, не должно превышать размер пула соединений.

- Если в приложении есть пул потоков (например, веб-контейнер с 200 потоками), а пул соединений настроен на 20, то:
  - 200 потоков будут одновременно пытаться получить соединение.
  - 20 получат соединение и будут работать, остальные 180 будут **блокироваться в очереди пула соединений** (ждать `connectionTimeout`).
  - Это может привести к исчерпанию ресурсов: потоки ждут, накапливаются задачи, может случиться OOME или падение сервера.

**Правило:**

```
Размер пула потоков (или число одновременных запросов) ≤ Размер пула соединений
```

То есть максимальное количество параллельных запросов, которым нужно соединение, не должно превышать `maximumPoolSize`.

**Пример хорошей настройки:**

```yaml
# Для синхронного веб-приложения с Tomcat
server.tomcat.threads.max: 50        # максимум 50 рабочих потоков
spring.datasource.hikari.maximum-pool-size: 50   # столько же соединений
```

**Но если у вас асинхронный фреймворк (WebFlux, Vert.x) — соединения должны быть неблокирующими, а пулы не нужны в традиционном виде (используют неблокирующие драйверы).**

## 7. Типичные проблемы и их решения

### Проблема 1: Утечка соединений (Connection Leak)

**Симптомы:**
- Количество активных соединений в пуле со временем растёт, доходит до `maximumPoolSize`.
- Приложение зависает, все ждут соединения.
- В логах HikariCP: `Connection leak detection` сообщения.

**Причина:** Приложение вызвало `getConnection()`, но не вызвало `close()` (или не закрыло в `finally`). В результате соединение не вернулось в пул.

**Исправление:**
```java
// Плохо
Connection conn = dataSource.getConnection();
Statement st = conn.createStatement();
st.executeQuery("SELECT ...");
// Нет close() — утечка!

// Хорошо (try-with-resources, Java 7+)
try (Connection conn = dataSource.getConnection();
     Statement st = conn.createStatement();
     ResultSet rs = st.executeQuery("SELECT ...")) {
    // работа
} // автоматически закрываются в обратном порядке

// Если не можете использовать try-with-resources (например, Spring без этого)
try {
    Connection conn = dataSource.getConnection();
    // ...
} finally {
    if (conn != null) conn.close(); // обязательно
}
```

**В Spring с JdbcTemplate/TxManager** утечки обычно нет, но если вы вручную получаете соединение через `DataSourceUtils`, нужно также правильно освобождать.

#### Проблема 2: Исчерпание пула (Pool Exhaustion)

**Симптомы:** В логах `Timeout waiting for connection from pool` (или аналогичное у Hikari). Приложение зависает.

**Возможные причины:**
1. Утечка соединений (см. выше).
2. Максимальный размер пула слишком мал для пиковой нагрузки.
3. Транзакции очень долгие, соединения удерживаются минутами.
4. Соединения не возвращаются из-за бесконечных запросов или блокировок в БД.

**Решения:**
- Увеличить `maximumPoolSize` после анализа (но не бездумно — есть риск перегрузить БД).
- Оптимизировать длительность транзакций (не держите соединение во время долгих вычислений, не делайте сетевых вызовов внутри транзакций).
- Добавить мониторинг активных соединений (JMX метрики).

#### Проблема 3: Перегрузка БД (Too many connections)

**Симптомы:** БД выдаёт ошибку `max_connections exceeded`, приложение падает.

**Причина:** Сумма `maximumPoolSize` по всем приложениям, подключённым к БД, превышает настройку `max_connections` на сервере БД.

**Решение:**
- На сервере БД: `SHOW max_connections` (PostgreSQL) или аналоги. Увеличить, если позволяет память.
- На стороне приложения: уменьшить `maximumPoolSize`.
- Для микросервисов — синхронизировать на уровне конфигурации.

#### Проблема 4: Соединения не валидируются (stale connections)

**Симптомы:** Приложение работает нормально, но через некоторое время (после простоя) начинаются исключения (например, `Connection closed`, `SocketException`, `The last packet sent successfully to the server was 0 milliseconds ago`).

**Причина:** Сетевое оборудование или БД закрывает неактивные соединения, а пул этого не знает и выдаёт "мёртвое" соединение.

**Исправление:**
```yaml
spring.datasource.hikari.connection-test-query: SELECT 1
spring.datasource.hikari.validation-timeout: 5000
spring.datasource.hikari.leak-detection-threshold: 60000
```

Также настроить `maxLifetime` меньше, чем таймаут неактивного соединения на стороне БД (например, в PostgreSQL `tcp_keepalives_idle`, в MySQL `wait_timeout`).

### 8. Мониторинг пула соединений

HikariCP предоставляет метрики через JMX и через поддержку Micrometer (для Prometheus).

**JMX включение:**
```yaml
spring.datasource.hikari.register-mbeans: true
```
Затем через JConsole или VisualVM смотреть:
- `HikariPool.n`: `ActiveConnections`, `IdleConnections`, `TotalConnections`, `PendingThreads` (ждущие очереди).

**Micrometer + Prometheus (Spring Boot Actuator):**
```yaml
management.metrics.export.prometheus.enabled: true
management.endpoints.web.exposure.include: metrics,prometheus
```
Доступные метрики:
- `hikaricp.connections.active`
- `hikaricp.connections.idle`
- `hikaricp.connections.max`
- `hikaricp.connections.pending`
- `hikaricp.connections.timeout` (количество таймаутов)

### 9. Как избежать проблем: чеклист

- [ ] Всегда закрывайте ресурсы (`Connection`, `Statement`, `ResultSet`) в `finally` или try-with-resources.
- [ ] Настройте `leakDetectionThreshold` (например, 30-60 секунд), чтобы логи помогли найти утечки при разработке.
- [ ] Установите `maximumPoolSize` не выше, чем разрешённая нагрузка на БД (`max_connections` всех приложений вместе).
- [ ] Установите `maxLifetime` меньше `wait_timeout` БД.
- [ ] Включите метрики и мониторинг (Grafana, алерты при `pending > 0` длительное время).
- [ ] Используйте правильную стратегию пула: для синхронных приложений — `maximumPoolSize` ≈ количество одновременных запросов. Для асинхронных — не пул, а клиент с ограничением.
- [ ] Тестируйте код на утечки с помощью инструментов (например, тест, который делает 10 000 запросов и проверяет, что пул возвращается к `minimumIdle`).

### 10. Сравнение с пулом потоков (общее и различия)

| Аспект | Пул потоков (`ThreadPoolExecutor`) | Пул соединений (`HikariCP`) |
|--------|-------------------------------------|-------------------------------|
| Ресурс | Потоки ОС (дорогие) | Соединения с БД (тоже дорогие) |
| Oversubscription | Плохо для CPU-bound | Плохо для БД (перегрузка) |
| Типичная ошибка | Слишком много потоков → `unable to create new native thread` | Слишком много соединений → `too many connections` в БД |
| Утечка ресурса | Поток не завершён (висячий) | Соединение не закрыто (утечка) |
| Обработка таймаутов | Есть `keepAliveTime` | Есть `idleTimeout`, `connectionTimeout` |
| Мониторинг | `getActiveCount()`, `getQueue().size()` | `getActiveConnections()`, `getTotalConnections()` |

**Связь:** оба пула управляют конечными ресурсами. Если пул потоков создаст 1000 потоков, и каждый попытается получить соединение, но пул соединений ограничен 20 — 980 потоков будут бессмысленно ждать и блокировать память.

### 11. Итог: минимальная хорошая конфигурация для большинства приложений

```yaml
spring:
  datasource:
    url: jdbc:postgresql://dbhost:5432/appdb
    username: app
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      leak-detection-threshold: 60000
      validation-timeout: 5000
      connection-test-query: SELECT 1
      auto-commit: false
      pool-name: AppHikariPool
```

**Для высоконагруженных приложений** делайте связку: пул соединений ≈ числу ядер × 2..4, но не более 100 (зависит от БД). Используйте профилирование для точной настройки.

Пулы соединений — не магия, их нужно понимать, настраивать и мониторить. Но без них ни одно серьёзное приложение с БД не проживёт.
