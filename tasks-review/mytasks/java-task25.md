# Трасировка

## Трассировка в микросервисах (Distributed Tracing)

Вы перешли к ключевой теме наблюдаемости (observability) в микросервисах. Трассировка (tracing) — это метод отслеживания запроса по всем сервисам, через которые он проходит. В отличие от метрик (средние значения) и логов (отдельные события), трассировка даёт целостную картину **одного конкретного запроса** или транзакции.

### 1. Зачем нужна трассировка в микросервисах?

В монолите вы можете просто посмотреть логи одного процесса. В микросервисах один запрос клиента может пройти через 5-20 сервисов (API Gateway → Auth Service → Order Service → Inventory Service → Payment Service → Notification Service). Без трассировки ответить на вопросы невероятно сложно:

- Какой сервис вызвал задержку?
- Где произошла ошибка (какой именно метод, какой сервис)?
- Был ли запрос к БД? Сколько времени он занял?
- Почему цепочка вызовов привела к OOME в одном из сервисов?

**Трассировка даёт:**
- Идентификатор запроса (`traceId`), который передаётся через все сервисы.
- Иерархию вызовов (родительские и дочерние спаны).
- Время каждого шага.
- Атрибуты (URL, HTTP-статус, имя БД, размер ответа, ошибки).

### 2. Ключевые понятия трассировки (по стандарту OpenTelemetry)

| Термин | Описание |
|--------|-----------|
| **Trace** | Полный путь запроса через систему. Представляет собой дерево **спанов** (spans). |
| **Span** | Единица работы: один логический шаг (например, вызов HTTP, запрос в БД, обработка в контроллере). Имеет имя, время начала, длительность, атрибуты, статус (OK/ERROR). |
| **TraceId** | Глобальный уникальный идентификатор трассировки (обычно 16 байт, 32 hex). Одинаков для всех спанов одного запроса. |
| **SpanId** | Идентификатор конкретного спана. |
| **ParentSpanId** | Ссылка на родительский спан (чтобы построить дерево). |
| **Baggage** | Дополнительные пары ключ-значение, передаваемые между сервисами (например, `userId`, `tenant`). |

### 3. Архитектура сбора трассировок

```
[Приложение A] ---(span)--> [Trace Collector (Jaeger/Zipkin)] ---> [Storage (Elasticsearch)]
      ↑                              ↓
      └---- (traceId, spanId) ------┘
```

Клиентский код (с помощью библиотеки) отправляет спаны на агент или непосредственно в коллектор. Коллектор агрегирует, сохраняет и даёт UI для просмотра.

### 4. Стандарты и реализации

| Инструмент | Статус | Особенности |
|------------|--------|-------------|
| **Jaeger** (CNCF) | Популярный, production-ready | Поддерживает OpenTracing, OpenTelemetry, имеет хороший UI, интеграцию с K8s. |
| **Zipkin** | Популярный, более простой | Старше, используется в Spring Cloud Sleuth (заменён на Micrometer Tracing). |
| **OpenTelemetry** | Единый стандарт (CNCF) | Объединяет метрики, трейсы, логи. Рекомендуется для новых проектов. |

**Spring Boot 3+** использует **Micrometer Tracing** (замена Spring Cloud Sleuth), который работает поверх OpenTelemetry или Brave (Zipkin).

### 5. Реализация в Spring Boot с Micrometer Tracing

#### Зависимости (Maven):

```xml
<!-- Micrometer Tracing с реализацией для Zipkin или Jaeger -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>
<!-- Опционально: авто-конфигурация для отправки трассировок в Zipkin/Jaeger -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-integration-test</artifactId>
    <scope>test</scope>
</dependency>

<!-- Для отправки на OpenTelemetry коллектор (альтернатива) -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
<dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-exporter-otlp</artifactId>
</dependency>
```

#### Конфигурация в `application.yml`:

```yaml
management:
  tracing:
    sampling:
      probability: 1.0   # 100% трассировок (в проде обычно 0.1-0.01, чтобы не перегружать)
    propagation:
      type: w3c          # или b3 (Zipkin)
  zipkin:
    tracing:
      endpoint: http://jaeger-collector:14268/api/v2/spans  # для Jaeger (использует Zipkin формат)
  # Если используете OpenTelemetry OTLP:
  otlp:
    tracing:
      endpoint: http://otel-collector:4318/v1/traces
```

#### Автоматическая инструментация (Spring Boot делает сама):
- Все входящие HTTP-запросы (Spring MVC, WebFlux)
- Исходящие HTTP запросы (RestTemplate, WebClient, Feign)
- Вызовы в БД (через DataSource, R2DBC)
- Запросы в Redis, Kafka, RabbitMQ
- Запланированные задачи (`@Scheduled`)

#### Ручное создание спанов:

```java
import io.micrometer.tracing.Tracer;
import io.micrometer.tracing.Span;

@Service
public class BusinessService {
    @Autowired
    private Tracer tracer;
    
    public void doComplexOperation() {
        Span span = tracer.nextSpan().name("custom-operation").start();
        try (Tracer.SpanInScope ws = tracer.withSpan(span)) {
            span.tag("user.id", "12345");
            // бизнес-логика
        } catch (Exception e) {
            span.error(e);
            throw e;
        } finally {
            span.end();
        }
    }
}
```

### 6. Интеграция с Kubernetes и Actuator

В K8s обычно запускают **Jaeger Operator** или **Helm-чарт Jaeger**. Приложение должно знать, куда слать спаны. Лучше всего использовать **service discovery** или переменную окружения:

```yaml
# Deployment.yaml
env:
- name: MANAGEMENT_ZIPKIN_TRACING_ENDPOINT
  value: http://jaeger-collector.observability.svc.cluster.local:14268/api/v2/spans
```

Actuator **не даёт эндпоинт для просмотра трасс** (это не его задача), но Actuator может показывать метрики (`/actuator/metrics/http.server.requests`), которые полезны в связке с трассировкой.

**Связь трассировки с метриками и логами:**
- **Логи** должны включать `traceId` и `spanId` (через MDC). Spring Boot автоматически добавляет их при использовании `micrometer-tracing`. Настройка в `logging.pattern.level`:
```yaml
logging:
  pattern:
    level: "%5p [${spring.application.name:},%X{traceId:-},%X{spanId:-}]"
```
- **Метрики** могут быть расщеплены по `traceId` (редко, обычно агрегируются).

### 7. Как трассировка помогает бороться с OOME и проблемами пулов потоков/соединений

Представьте: в микросервисе OrderService возникает `OutOfMemoryError`. Heap dump покажет, что много объектов `HttpConnection` или `DatabaseConnection`, но из какого запроса? Без трассировки **непонятно**.

С трассировкой:
- В логах (перед OOME) виден `traceId` последнего запроса.
- Идёте в Jaeger, ищете этот `traceId`.
- Вы видите: запрос от пользователя X вызвал цепочку: API Gateway → OrderService → InventoryService (который вернул огромный список) → OrderService начал кешировать этот список в памяти → утечка.
- Также видите, что в OrderService воркер из пула потоков обрабатывал этот запрос, но не освободил соединение с БД (долго держал транзакцию), и пул соединений исчерпался.

**Конкретные сценарии:**

| Проблема | Что видим в трассировке |
|----------|--------------------------|
| Утечка соединений с БД | Спан БД занимает часы (соединение не закрыто). Видно, какой endpoint не вызвал `close()`. |
| Пул потоков исчерпан | Длинные очереди на пуле потоков (можно посмотреть по метрикам, но трассы покажут, какие запросы долго висят). |
| OOME из-за кеша | Множество вызовов, каждый добавляет данные в статическую коллекцию. В дампе видно, но трейс поможет найти "размножитель" (например, баг в цикле). |
| Неисправный downstream | Спан вызова другого сервиса имеет ошибку (HTTP 500, таймаут). Это может объяснить, почему ваш сервис накапливает задачи или держит ресурсы. |

### 8. Мониторинг производительности запросов

С помощью трассировки можно найти **медленные запросы**. Jaeger UI позволяет:

- Искать по `serviceName`, `operationName`, `tags`, `duration`.
- Смотреть распределение длительностей (percentile).
- Находить longest trace.
- Анализировать зависимые вызовы (система покажет граф).

**Пример поиска:** найти все запросы к `PaymentService`, которые завершились ошибкой и длились больше 2 секунд. Вы увидите точную причину (база данных, сеть, таймаут).

### 9. Полезные практики

- **Семплирование (sampling):** в проде не нужно трассировать каждый запрос (будет слишком много данных). Обычно `probability=0.1` (10% запросов). Или адаптивное семплирование (ошибки и медленные трейсы сохраняются всегда).
- **Пользовательские теги (tags):** добавляйте ключевые поля: `orderId`, `userId`, `error.message`. Это позволит быстро искать релевантные трейсы.
- **Корреляция с логами:** убедитесь, что ваш log aggregator (ELK, Loki) индексирует `traceId`. Вы можете найти логи конкретного запроса, перейдя из UI трассировки.
- **Не перегружайте спаны:** один спан должен делать одно логическое действие. Не создавайте тысячи спанов на каждый цикл.
- **Интеграция с ошибками:** если возникает исключение, записывайте в спан `status=ERROR` и добавляйте `exception` атрибут.

### 10. Пример: отладка OOME с помощью трассировки

1. Приложение падает с OOME, настроен `HeapDumpOnOutOfMemoryError`. Дамп получен.
2. Смотрим в логах (через ELK) последние записи перед падением. Видим `traceId=abc123`.
3. Идём в Jaeger, ищем `abc123`. Открываем трассу:
   - Спан `POST /api/orders` длится 10 секунд.
   - Внутри есть дочерний спан `db.query` длительностью 100мс.
   - Еще спан `inventoryClient.getProducts` длится 9.8 секунд (таймаут).
4. Дополнительные атрибуты спана: `inventoryClient.responseSize=500MB` (inventory вернул гигантский JSON).
5. Приложение пыталось загрузить этот ответ целиком в память, что привело к OOME.
6. Исправление: пагинация или стриминг ответа от inventory service, либо ограничение в клиенте.

### 11. Сравнение: трассировка vs метрики vs логи

| Аспект | Метрики | Логи | Трассировка |
|--------|---------|------|--------------|
| Что даёт | Агрегаты (средние, перцентили) | Отдельные события | Полный путь одного запроса |
| Пример | 95й процентиль времени ответа 500ms | "User 123 не авторизован" | Весь путь запроса через 5 сервисов с таймингами |
| Память/процессор | Мало | Много (индексация) | Умеренно (зависит от семплирования) |
| Для отладки багов | Плохо (усреднение) | Хорошо (нужен traceId в логах) | Отлично |

### 12. Резюме: что нужно сделать для внедрения трассировки

1. **Добавить зависимости** Micrometer Tracing + реализацию (Zipkin/Brave или OpenTelemetry).
2. **Настроить endpoint коллектора** (Jaeger или Zipkin) через конфигурацию.
3. **Включить пересылку traceId в логи**.
4. **Запустить коллектор** в кластере K8s (Jaeger Operator или Helm).
5. **Включить семплирование** (0.1 в проде, 1.0 на staging).
6. **Добавить кастомные теги** в ключевые места (ID заказа, юзера, ошибки).
7. **Обучить команду** пользоваться UI Jaeger для поиска проблем.

### 13. Пример полной настройки с K8s

**values микросервиса (Helm):**
```yaml
env:
- name: MANAGEMENT_TRACING_SAMPLING_PROBABILITY
  value: "0.1"
- name: MANAGEMENT_ZIPKIN_TRACING_ENDPOINT
  value: "http://jaeger-collector.monitoring:14268/api/v2/spans"
- name: LOGGING_PATTERN_LEVEL
  value: "%5p [${spring.application.name:},%X{traceId:-},%X{spanId:-}]"
```

**Развертывание Jaeger через Istio (если используете):**
```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: mesh-default
spec:
  tracing:
    - providers:
        - name: jaeger
      randomSamplingPercentage: 10.0
```

**Без Istio:** достаточно запустить Jaeger через официальный оператор.

### 14. Что не может трассировка

- Она не заменит heap dump для анализа утечек памяти.
- Не покажет использование CPU профилирования.
- Не даст точные метрики в реальном времени (это роль Prometheus).
- Требует внедрения во **всех** сервисах, чтобы цепочка была полной.

### Итог

Трассировка — это рентгеновский снимок вашего распределённого запроса. Без неё вы слепы в микросервисах. В сочетании с дампами heap (для OOME) и пулами соединений вы получаете полную картину: **что, где и почему пошло не так**. Actuator здесь помогает предоставить метрики и здоровье, а трассировка — понять динамику запросов.
