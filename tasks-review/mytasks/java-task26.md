# Процесс разбора инцидентов(например с JVM)

## Процесс разбора инцидентов с JVM в микросервисах

В микросервисной архитектуре инцидент (например, `OutOfMemoryError`, длительный GC pause, резкий рост CPU, падение подов) требует **системного подхода**, потому что проблема может быть локализована в одном сервисе, но иметь корень в другом (например, утечка памяти из-за слишком большого ответа от зависимого сервиса). Разбор инцидента — это цикл: обнаружение → локализация → углублённый анализ → принятие мер → постмортем.

### 1. Обнаружение инцидента (раннее оповещение)

В микросервисах вы **никогда не узнаете о проблеме первым от пользователя** (если только у вас нет хорошего мониторинга). Используйте:

- **Метрики** (Prometheus + Grafana):
  - JVM метрики: использование heap (байты и процент), метаспейс, GC время и частота, количество потоков, время работы приложения.
  - Системные метрики: CPU usage, память pods, количество рестартов контейнера.
  - Бизнес-метрики: длительность ответа, количество ошибок HTTP 5xx.
- **Логи** с агрегацией (ELK, Loki):
  - Сбор всех `java.lang.OutOfMemoryError`, `GC overhead limit exceeded`, `Unable to create new native thread`.
  - Уровень ERROR с указанием сервиса и `traceId`.
- **Трассировка** (Jaeger/Zipkin):
  - Аномальное увеличение длительности запросов и всплеск ошибок могут быть триггером.
- **Kubernetes события**:
  - Pod в статусе `OOMKilled` (превышение лимита памяти контейнера, не путать с JVM OOME!)
  - CrashLoopBackOff, liveness probe failure.

**Пример алерта (Prometheus):**
```yaml
- alert: JVMHeapUsageHigh
  expr: sum(jvm_memory_used_bytes{area="heap"}) / sum(jvm_memory_max_bytes{area="heap"}) > 0.85
  for: 5m
  annotations:
    summary: "Сервис {{ $labels.service }} приближается к OOME"
```

### 2. Локализация: какой сервис и какой узел?

В распределённой системе нужно найти **какой именно под** и **какое приложение** виновато.

**Шаги:**
- Посмотреть дашборд (Grafana) по всем сервисам: у какого сервиса резко вырос heap usage, частота GC, количество перезапусков.
- В K8s: `kubectl get pods -n <namespace>` — найти поды в статусе `Error`, `CrashLoopBackOff`, `OOMKilled`.
- Проверить логи: `kubectl logs <pod> --previous` (если под перезапустился).
- Если настроены метки (labels), то использовать `kubectl describe pod` для получения причины последнего завершения.

**Отличить OOME от OOMKill:**
- `OutOfMemoryError` внутри JVM → приложение падает с ошибкой Java, но контейнер может не быть убит (если JVM сработала раньше лимита контейнера). В логах будет `java.lang.OutOfMemoryError`.
- `OOMKilled` – контейнер превысил memory limit, и его убил cgroup. В логах кубера: `The container was OOMKilled`. Это значит, лимит памяти контейнера меньше, чем потребовала JVM (например, лимит 1Gi, а JVM запросила 1.5Gi через `-Xmx`). В этом случае дамп heap не создаётся (убийство внешнее).

### 3. Сбор данных для анализа (до перезапуска)

Если контейнер ещё жив или у вас есть persistent volume для дампов:

- **Heap dump** (при `-XX:+HeapDumpOnOutOfMemoryError` он уже должен лежать в указанной папке). Скопировать на машину с MAT.
- **Thread dump** (jstack, или через Actuator `/actuator/threaddump`).
- **GC logs** (если включены: `-Xloggc:/path/gc.log -XX:+PrintGCDetails`).
- **Метрики за последние часы** (Grafana screenshot).
- **Логи всех сервисов, которые взаимодействуют с проблемным** (по трассировке).

**Важно:** в микросервисах под может быть удалён при перезапуске (если не настроены дампы на внешнем хранилище). Настройте `heapDumpPath` на том, который не удаляется при рестарте пода (например, пустую пустоту? в K8s лучше использовать PersistentVolumeClaim или Sidecar-контейнер для выгрузки дампов в S3).

### 4. Анализ первопричины

Работает та же методология, что и для монолита, но с поправкой на распределённость.

#### Типичные JVM-инциденты в микросервисах:

| Симптом | Что проверяем | Инструмент |
|---------|---------------|-------------|
| OOME Java heap space | Heap dump → biggest objects → почему не освобождаются | Eclipse MAT |
| OOME Metaspace | Количество загруженных классов, ClassLoader утечки | MAT / jstat |
| OOME Direct buffer memory | Number of DirectByteBuffer, использование off-heap | MAT + ленты трассировки |
| OOME unable to create new native thread | Количество потоков, лимиты ulimit, пулы потоков | thread dump, `ps -eLf`, метрики |
| Длительные GC паузы | GC logs, использование памяти, размеры регионов G1 | GCViewer, GCEasy |
| Высокий CPU | Профилирование (async-profiler), thread dump | JMC, VisualVM |

#### Особенность микросервисов: как трассировка помогает анализу

- В thread dump вы можете увидеть стеки потоков, обрабатывающих запросы. Если много потоков ждут ответа от другого сервиса (например, HTTP клиент), то трассировка (Jaeger) покажет, какой именно запрос «висит».
- Heap dump может содержать тысячи объектов, удерживаемых из-за не закрытых HTTP клиентов или слушателей. По `traceId` в аттрибутах объектов (например, в слаге) можно найти исходный запрос.

**Примерный процесс углублённого анализа OOME:**
1. Открыть heap dump в MAT.
2. Leak Suspects Report → увидеть крупный объект (например, `ArrayList` с миллионом записей).
3. Path to GC Roots → выяснить, что этот список хранится в статическом поле `RequestCache.cache`.
4. Посмотреть в коде: этот кеш используется для хранения ответов из сервиса `product-service`.
5. Запросить трассировку (Jaeger) по одному из traceId, найденных в объектах кеша (если логировали).
6. Обнаружить, что `product-service` при определённых условиях возвращает очень большой объём данных (>10MB), и кеш не имеет политики вытеснения.
7. Причина: не сработала пагинация из-за бага в `product-service`. Исправление: добавить ограничение на сторону клиента или исправить сервис.

### 5. Устранение инцидента (remediation)

Действия в зависимости от характера:

- **Краткосрочные**:
  - Увеличить memory limit контейнера (если OOMKilled) и соответствующий `-Xmx`.
  - Увеличить `maxPoolSize` пула соединений или потоков.
  - Перезапустить проблемный под (если утечка временная).
  - Отключить кеш или уменьшить TTL.
- **Долгосрочные**:
  - Исправить утечку памяти (закрыть ресурсы, добавить эвикшен).
  - Внедрить автоматическое профилирование в CI.
  - Улучшить мониторинг (алерты на приближение к лимиту).

**В микросервисах можно применять паттерны «Circuit Breaker» и «Backpressure»:** если вы видите, что downstream сервис возвращает слишком много данных, лучше разорвать цепь (выбросить исключение) и обработать ошибку, чем загружать память.

### 6. Постмортем (post-mortem) и предотвращение

После инцидента нужно провести **blameless post-mortem** (без обвинений) с ответами:
- Что произошло? (таймлайн, симптомы, длительность)
- Почему это произошло? (первопричина)
- Как мы обнаружили? (сработал ли алерт, какой задержка)
- Что сделали для восстановления? (вручную/автоматически)
- Что мешало избежать? (отсутствие тестов, не хватало instrumentation)
- Действия для предотвращения повторения (fix, тест, изменение архитектуры, настройки).

**Пример действий после OOME:**
- Добавить интеграционный тест, который эмулирует большой ответ product-service и проверяет, что кеш не разрастается.
- Настроить `maximumSize` для кеша через Caffeine.
- Включить `leakDetectionThreshold` в пуле соединений.
- Добавить метрику `cache.size` в Prometheus.

### 7. Инструменты для разбора инцидентов в микросервисах

| Уровень | Инструмент | Назначение |
|---------|------------|------------|
| Мониторинг | Prometheus + Grafana | Сбор метрик, алерты, дашборды (JVM, K8s) |
| Логи | ELK (Elasticsearch, Logstash, Kibana) или Loki | Агрегация логов, поиск по traceId, `OutOfMemoryError` |
| Трассировка | Jaeger, Zipkin | Поиск по traceId, анализ длительности, ошибок |
| Дампы | Eclipse MAT, VisualVM, YourKit | Анализ heap dump (утечки) |
| Профилирование | async-profiler, JMC | CPU профилирование, выделение памяти |
| K8s события | `kubectl describe`, `k9s`, Lens | Статусы подов, события OOMKilled |

### 8. Процесс (checklist) на примере OOME

1. **Оповещение** – Prometheus алерт ‘JVMHeapUsageHigh’ или ‘PodRestartsHigh’.
2. **Смотрю в Grafana** – обнаруживаю, что сервис `payment-service` перезапускается каждые 5 минут.
3. **Смотрю логи (Kibana)** – фильтр по `service=payment-service` и уровню ERROR → вижу `java.lang.OutOfMemoryError: Java heap space`.
4. **Проверяю под** – `kubectl describe pod payment-service-xyz` – причина завершения: `Error`, exit code 137 (часто OOMKilled, но с OOME он может быть 1, зависит от скрипта). Вижу, что в конфигурации контейнера нет volume для дампов.
5. **Меняю временно конфиг** – увеличиваю `memory.limit` на 2Gi и `-Xmx` на 1.5Gi, перезапускаю.
6. **Получаю дамп** – после настройки `HeapDumpOnOutOfMemoryError` с `heapDumpPath` на PV, следующий OOME даст дамп.
7. **Анализ дампа в MAT** – обнаруживаю, что `PaymentProcessor` хранит в статическом списке все транзакции за последний час (100 млн).
8. **Причина** – баг в коде: список не очищается после обработки.
9. **Исправление** – ограничиваю размер списка и добавляю `clear` после обработки.
10. **Развертка CI/CD** – проходит тесты, деплоится. Инцидент закрыт.
11. **Постмортем** – добавляем тест на утечку, увеличиваем память? не нужно. Изменяем настройки дампов, добавляем метрику размера этого списка.

### 9. Особенности распределённого разбора (Architecture)

- **Стандартизация** – все сервисы должны использовать одинаковую библиотеку наблюдаемости (один формат логов, один trace propagation).
- **Централизованный доступ** – инженер должен иметь доступ к дашбордам, логам, трассировке из одного окна (Grafana с интегрированным Loki и Tempo).
- **Автоматизация сбора evidence** – при срабатывании алерта можно запускать сбор thread dump, heap dump, метрик из всех подов данного сервиса (через оператор).

### Итог

Процесс разбора JVM-инцидентов в микросервисах включает **раннее обнаружение** (метрики, алерты), **локализацию** (по какому сервису, поду), **сбор артефактов** (heap dump, логи, трассировка), **анализ** (MAT, трассировка, профилирование), **принятие мер** и **постмортем**. Ключевое отличие от монолита — необходимость связывать состояние JVM одного сервиса с контекстом запроса (`traceId`) и поведением других сервисов. Без наблюдаемости (observability) разбор становится гаданием.
