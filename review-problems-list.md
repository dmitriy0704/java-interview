# Памятка на код-ревью:

На код-ревью принято разделять замечания по степени критичности. Это помогает
автору кода понять, что нужно исправить немедленно, а что — просто совет на
будущее.
Вот основная классификация проблем:

## 1. Критические проблемы (Blockers / Major)

Это ошибки, которые «сломают» приложение в будущем или приведут к потере данных.

* Логические ошибки: Неправильные условия в if, неверные расчеты.
* Проблемы с транзакциями: Отсутствие @Transactional там, где это нужно (или его
  наличие в методах, которые вызываются внутри того же класса).
* Производительность: Проблема N+1 (запросы в цикле), отсутствие индексов под
  частые запросы, сохранение миллионов записей без батчинга.
* Безопасность: Хардкод паролей, отсутствие проверки прав доступа (
  SecurityContext), уязвимость к SQL-инъекциям.
* Утечки ресурсов: Недозакрытые потоки (Streams), соединения или кэши, которые
  забивают память.

## 2. Средние проблемы (Medium / Major Design)

Код работает, но он написан «грязно», и его будет трудно поддерживать.

* Нарушение принципов SOLID/DRY: Слишком большие классы (God Object),
  дублирование кода в разных сервисах.
* Неправильное использование фреймворка: Например, создание бинов через new,
  внедрение зависимостей не через конструктор, а через @Autowired на полях.
* Жесткая связность (Tight Coupling): Использование ApplicationContext.getBean()
  вместо нормального внедрения зависимостей.
* Плохая обработка ошибок: Пустые блоки catch, логгирование ошибок без
  пробрасывания дальше или использование e.printStackTrace() вместо логгера.

## 3. Минорные проблемы (Minor / Style)

Это вопросы стиля, которые не влияют на работу программы, но влияют на «чистоту»
проекта.

* Именование: Непонятные переменные (a, list1), несоблюдение camelCase.
* Магические числа и строки: Использование String region = "Siberia" вместо
  констант или Enum.
* Избыточность: Лишние аннотации (например, @Data там, где достаточно @Getter),
  неиспользуемые импорты или переменные.
* Комментарии: Старый закомментированный код (его нужно просто удалять) или
  комментарии, которые просто пересказывают код.

## 4. Образовательные советы (Nitpicks / Hints)

Это советы «на вырост», которые не обязательны к исполнению.

* Синтаксический сахар: «Здесь можно использовать Stream API вместо цикла», или
  «В Java 17 этот switch можно написать короче».

# Для Spring Boot приложений

Вот ваш чек-лист из 5 пунктов, которые «отлавливают» 80% проблем в Spring Boot
сервисах. Сохраните его для следующего ревью:

## 1. Транзакционный контекст (The @Transactional check)

* Снаружи, а не внутри: Метод с @Transactional вызывается из другого класса? (
  Если метод помечен аннотацией, но вызывается из того же класса через
  this.method(), транзакция не откроется).
* RuntimeException: Выбрасываются ли внутри только RuntimeException? (По
  умолчанию Checked exceptions не откатывают транзакцию).

## 2. Эффективность БД (The N+1 check)

* Запросы в циклах: Есть ли внутри for или stream.forEach вызовы репозитория или
  других сервисов, идущих в базу?
* Массовые операции: Используется ли saveAll() для списков вместо одиночного
  save()?
* Ленивая загрузка: Нет ли обращения к полям связанных сущностей, которые могут
  вызвать скрытый запрос в базу?

## 3. Чистота зависимостей (The DI check)

* Конструктор: Внедряются ли зависимости через final поля и конструктор (или
  @RequiredArgsConstructor)?
* Никаких new: Не создаются ли вручную экземпляры репозиториев, сервисов или
  других бинов?
* Отсутствие Context: Не используется ли ApplicationContext для ручного поиска
  бинов по имени?

## 4. Качество данных и типов (The Type check)

* Enums vs Strings: Используются ли перечисления (Enum) для статусов, типов и
  категорий вместо строк?
* Optional: Обрабатывается ли результат findById (и других Optional) через
  .orElseThrow() или .ifPresent()? Нет ли прямого вызова .get() без проверки?

## 5. Архитектурная чистота (The Logic check)

* DTO vs Entity: Не протекают ли сущности базы данных (@Entity) напрямую в API
  контроллеры? (Хорошая практика — мапить их в DTO).
* Валидация: Проверяются ли входные параметры на null или пустые значения до
  начала обработки?
